§ III

Per-step detail

What each phase is for
Q
QuestionsPhase one
skill · rpi:create-research-questions
Resolve ambiguity before R wastes effort.
  • Tiny input, errors compound downstream
  • Bump effort when reqs are messy
  • Review the questions — don't outsource judgment
AvoidLetting Q smuggle in design decisions.
R
ResearchPhase two
skill · rpi:create-research
Evidence collection over complex codebases.
  • Reasoning-bound — Opus 4.7 leads SWE-bench Pro (64.3 vs 58.6)
  • gpt-5.5 low is sparse and dense — bump to high for synthesis
  • 1M Opus for big repos — but note 4.7's multi-needle retrieval regressed vs 4.6
AvoidCheap research as the foundation for everything downstream.
PRD
Product Req. DocPhase three
skill · rpi:create-prd
Decide product behavior, UX, and scope.
  • Mockup-driven — show, don't tell
  • Dialogue with you in the loop; one decision at a time
  • Lock UX before TDD opens architecture
AvoidSmuggling technical decisions into the product spec.
TDD
Technical Design DocPhase four
skill · rpi:create-tdd
Decide architecture, signatures, data flow.
  • References the PRD — don't restate it
  • Mermaid for flows; concrete type signatures for new APIs
  • Start in a fresh session after PRD; no autoadvance
AvoidVague code sketches. Sign types or skip.
S
StructurePhase five
skill · rpi:create-structure-outline
File layout and phase ordering given TDD's call.
  • Highest-leverage planning step — never cheap out
  • Small output, big consequence on rework cost
  • If S struggles, TDD didn't decide enough
AvoidDropping below high effort, ever.
P
PlanPhase six · often skipped
skill · rpi:create-plan (or skip via implement-outline)
Sequenced steps with validation criteria.
  • Often replaced by rpi:implement-outline — go S → I directly
  • Still worth the round-trip for high-risk or auth/migration work
  • Must include rollback / failure criteria when present
AvoidGenerating a plan that just paraphrases the outline.
I
ImplementPhase seven
skill · rpi:implement-outline · rpi:implement-plan
Execute the agreed outline or plan; report violations.
  • Codex/gpt-5.5 at high or xhigh — literal execution of a tight plan is its strength
  • Hashimoto on Codex xhigh: solved a 6-month bug in 45min where Opus failed
  • Contract: follow the doc; if wrong, stop and report
AvoidLetting I reinterpret or improvise on the structure.

The frame

Use 1M when many live relationships must be held simultaneously. Not when you want the model to think harder.

Failure modes

  • Stale assumptions anchor
  • Signal drowns in noise
  • Summarizing over deciding
  • Bad context preserved

Claude gotcha

opusplan does not get the 1M auto-upgrade. Use explicit --model opus for big-context planning.

§ VII

Core principles

Six axioms
01
Reasoning > recall on big repos. Cheap research compounds into bad outlines. Bump R first.
02
Split product from technical. PRD locks UX, TDD locks architecture. Don't blur them.
03
Speed enables iteration. Fast high-quality loops beat one slow perfect pass.
04
TDD and S are the floor. Plan is increasingly optional — outline → implement is preferred.
05
1M = working memory. Not more thinking. Hold relationships, don't replace reasoning.
06
Wanting 1M downstream = lossy upstream. Fix the handoff, not the context window.
§ VIII

A short list of don'ts

Common failures, named

Workflow

  • Skip PRD or TDD, then S inherits the open questions
  • Same dial across all seven steps
  • Bump everything when one step struggles
  • Run PRD and TDD in the same session — start TDD fresh

Model picks

  • Use gpt-5.5 low for research — sparse and dense
  • Use Opus for implementation of a tight plan — Codex executes literal plans faster
  • Trust Sonnet 4.6 to reason about large complex codebases
  • Use max as a default — overthinks; xhigh is the practical ceiling

Context & prompts

  • Trust Opus 4.7 1M for multi-needle retrieval — regressed vs 4.6
  • Hedge prompts to Opus 4.7 with "if possible" / "try to" — weakens instructions
  • Trust opusplan for big-context planning
  • Load 1M by default · use 1M to mask weak R
